DEPLOY NOTES — PCM-Hub
======================

==========================================================================
0) FILE DI PRODUZIONE
==========================================================================

Repository:
  docker-compose.yml          → stack di DEV (sul tuo PC)
  docker-compose.prod.yml     → stack di PROD (sulla VM, via Portainer)
  caddy/Caddyfile             → config reverse proxy + HTTPS automatico
  .env                        → SOLO sul tuo PC, gitignored, dev locale
  .env.example                → template (committato, niente segreti dentro)

Sulla VM:
  Niente .env, niente cartelle host, niente SSH. Tutto vive dentro
  Portainer e dentro volumi nominati Docker:
    - postgres_data    DB
    - caddy_data       certificati Let's Encrypt
    - caddy_config     stato Caddy
    - frontend_dist    SPA buildata (popolata dal service frontend_build)
    - pcm_backups      dump pg_dump (popolata dal service backup)

Le variabili d'ambiente stanno in Portainer (Stack → Environment variables).


==========================================================================
1) COSA SERVE SULLA VM
==========================================================================

- Docker + Docker Compose installati.
- Portainer.
- Il dominio hub.parametricomparison.unimore.it puntato all'IP della VM.
- Porte 80 e 443 raggiungibili da internet (per HTTPS automatico Caddy
  via Let's Encrypt).

Niente cartelle host da preparare a mano. Niente SSH richiesto. Tutto
quello che ti serve passa da Portainer (UI web + console dei container).

Reverse proxy: già configurato in caddy/Caddyfile. Caddy gira nello
stack stesso e gestisce sia HTTP→HTTPS sia il certificato Let's Encrypt
in automatico, senza intervento esterno. Quando arriverà il certificato
dei servizi informatici, vedi i commenti dentro caddy/Caddyfile per la
sostituzione (basta scommentare un blocco tls).

Frontend: la SPA viene buildata da un container "usa e getta" allo start
dello stack (service `frontend_build`, vedi sezione 4 e 6). La VM non
deve avere Node installato: ci pensa l'immagine node:20-alpine. La build
gira solo al primo start o a ogni "Pull and redeploy" e poi muore. Il
bundle finisce in un volume Docker `frontend_dist` che Caddy serve.


==========================================================================
2) GENERARE LE CREDENZIALI DI PRODUZIONE
==========================================================================

DEVI usare valori NUOVI in produzione, diversi da quelli del .env locale.
Le ragioni:
  - La SECRET_KEY del .env di dev è già passata sopra GitHub/chat → da
    considerare compromessa.
  - Le credenziali del DB di dev sono debolissime (pcm_user/pcm_password).

A) Generare una SECRET_KEY nuova (lunga, random):

    python -c "import secrets; print(secrets.token_urlsafe(48))"

   Copia il risultato. Niente spazi, niente apici intorno.

B) Inventare una password DB forte:
   minimo 20 caratteri random, niente parole comuni.
   Es. anche qui:
       python -c "import secrets; print(secrets.token_urlsafe(24))"

C) Decidere il nome utente DB e nome database (puoi tenerli, ma
   conviene cambiarli per non avere lo stesso schema di dev):
       POSTGRES_USER=pcm_prod
       POSTGRES_DB=pcm_hub_prod

D) Generare una password ADMIN forte (la userai per il primo login):

       python -c "import secrets; print(secrets.token_urlsafe(20))"

   Salvala nel password manager. Diventera' ADMIN_PASSWORD.
   ADMIN_EMAIL: scegli una tua email reale (per recovery, anche se
   il reset via email arrivera' solo dopo che avrai la chiave SMTP
   dei servizi informatici).


==========================================================================
3) CONFIGURARE PORTAINER (le variabili d'ambiente)
==========================================================================

In Portainer, quando crei lo stack, NON committare il file .env.
Imposta queste variabili nello stack, una per una:

    ENV=prod
    SECRET_KEY=<la chiave generata al punto 2.A>
    ACCESS_TOKEN_EXPIRE_MINUTES=30

    ADMIN_EMAIL=<la-tua-email>
    ADMIN_PASSWORD=<password generata al punto 2.D>

    POSTGRES_USER=<nuovo utente>
    POSTGRES_PASSWORD=<password generata al punto 2.B>
    POSTGRES_DB=<nome db>
    POSTGRES_HOST=db
    POSTGRES_PORT=5432

    DATABASE_URL=postgresql://<user>:<password>@db/<dbname>

    CORS_ORIGINS=https://hub.parametricomparison.unimore.it
    (se usi più domini: separali con virgola, NIENTE spazi)

NOTE IMPORTANTI:
  - ENV=prod abilita il fail-fast: se manca SECRET_KEY (o ADMIN_EMAIL,
    o ADMIN_PASSWORD) l'app NON parte. Questo e' voluto, evita di andare
    in prod con la chiave di dev o senza un primo admin valido.
  - ADMIN_EMAIL/ADMIN_PASSWORD vengono usate UNA SOLA VOLTA al primo
    avvio (tabella users vuota) per creare il primo admin. Dopo il
    primo login cambia la password dal pannello, ma lascia comunque
    le variabili in Portainer perche' a fine import bundle vengono
    riapplicate (vedi sezione 7-bis passo 8).
  - DATABASE_URL deve essere coerente con i 4 valori sopra. Se ne cambi
    uno cambia anche l'altro.
  - CORS_ORIGINS deve avere ESATTAMENTE il dominio da cui parte il
    browser. Niente wildcard "*" (rifiutato dal browser quando hai
    allow_credentials).
  - NON c'e' piu' FRONTEND_DIST_PATH: la SPA vive in un volume Docker
    nominato (`frontend_dist`) popolato dal container di build.


==========================================================================
4) docker-compose DI PRODUZIONE — GIA' PRONTO
==========================================================================

File: docker-compose.prod.yml (al root del progetto). Cinque servizi:

  db              postgres:16-alpine, non esposto sull'host.
  backend         FastAPI buildato dal Dockerfile, expose interno 8000.
  frontend_build  node:20-alpine, "usa e getta": npm ci + npm run build,
                  copia dist/ nel volume `frontend_dist`, esce a 0.
                  Ribuilda a ogni start dello stack.
  caddy           reverse proxy + HTTPS, monta `frontend_dist` su
                  /srv/dist. Aspetta che frontend_build sia uscito
                  con successo prima di partire.
  backup          postgres:16-alpine in modalita' loop infinito: ogni
                  24h fa pg_dump nel volume `pcm_backups`. Tiene gli
                  ultimi KEEP=3 dump.
  cleaner         docker:25-cli, prune giornaliero (NON tocca i volumi).

Differenze rispetto al docker-compose.yml di dev:
  - DB non esposto sull'host: solo rete docker interna.
  - Backend non esposto sull'host: usa `expose: "8000"`, raggiungibile
    SOLO da Caddy via rete docker.
  - Nessun bind-mount del codice: gira l'immagine buildata dal Dockerfile.
  - uvicorn senza `--reload`, con `--proxy-headers --forwarded-allow-ips=*`
    per leggere correttamente l'IP del client da Caddy (serve al rate-limit).
  - Caddy in front, gestisce 80 e 443 e fa HTTPS automatico Let's Encrypt.
  - Logging json-file con rotation 10MB x 3 file su tutti i servizi.

Le variabili d'ambiente NON sono lette da .env: vanno impostate in
Portainer come "Stack environment variables" (vedi sezione 3).


==========================================================================
5) PRIMA INIZIALIZZAZIONE DEL DATABASE
==========================================================================

La prima volta che avvii lo stack:

  a) Lo schema viene creato dalle migrazioni Alembic. Esegui da
     Portainer: Containers → pcm_backend → Console → /bin/bash →
     "alembic upgrade head".

  b) Il primo admin viene creato AUTOMATICAMENTE da
     services/admin_bootstrap.py la prima volta che il backend parte
     con la tabella users vuota. Usa ADMIN_EMAIL e ADMIN_PASSWORD dalle
     variabili d'ambiente (sezione 3). Dal secondo avvio in poi e'
     un no-op: la password che hai cambiato dal pannello NON viene
     sovrascritta.

  c) Per importare i dati storici (sito vecchio): logghi come admin,
     vai sulla pagina "Database Restore / Migration", carichi il
     bundle ZIP con wipe=true. L'endpoint /api/admin/migration richiede
     gia' admin, quindi il flusso e': bootstrap (b) → login → import.
     A fine import la password admin viene riscritta a ADMIN_PASSWORD
     (vedi 7-bis passo 8): cambia la password DOPO l'import, non prima.


==========================================================================
6) DEPLOY DEL FRONTEND
==========================================================================

ZERO comandi manuali. La build avviene dentro lo stack, su Portainer.

Come funziona:
  1. Portainer fa git clone del repo (o "Pull and redeploy").
  2. Lo stack parte. Il service `frontend_build` (container node:20-alpine)
     monta `./frontend` read-only, copia i sorgenti in /build, lancia
     `npm ci && npm run build`, copia `dist/` nel volume nominato
     `frontend_dist`, ed esce a 0.
  3. Caddy parte solo quando frontend_build e' uscito con successo
     (depends_on: service_completed_successfully). Monta `frontend_dist`
     su /srv/dist read-only e serve la SPA.
  4. Per fare un nuovo deploy del frontend: pushi su GitHub, poi su
     Portainer → Stacks → pcm-hub → "Pull and redeploy". Il container
     di build rifa' tutto in automatico.

Il frontend NON usa piu' VITE_API_URL: parla con il backend via path
relativi (`/api/*`, `/auth/*`) che Caddy proxa al container backend.
Stesso dominio = niente CORS, niente env da settare in build.

Costo: la prima build dura ~30-60s (npm ci scarica i pacchetti). I
redeploy successivi sono leggermente piu' veloci grazie alla cache
delle immagini Docker, ma se cambia frontend/package-lock.json
ricomincia da capo. Memoria di picco ~400 MB durante la build:
sui tuoi 8 GB e' tranquillo.

Verifica post-deploy: apri la console di pcm_caddy in Portainer e
guarda i log; deve aver fatto "served /index.html" alle prime
richieste. Se la SPA non carica:
  - Containers → pcm_frontend_build → Logs: deve aver finito con
    "[frontend_build] dist copiata in /out, esco." e exit code 0.
  - Containers → pcm_caddy → Console → ls /srv/dist/ : deve mostrare
    index.html, assets/, ecc.


==========================================================================
7) BACKUP DEL DATABASE
==========================================================================

Niente cron host, niente script da copiare via scp. Il backup e' fatto
dal service `backup` definito in docker-compose.prod.yml. Si avvia con
lo stack e fa:

  - pg_dump del DB di prod ogni 24h verso /backups (volume `pcm_backups`).
  - Tiene gli ultimi 3 dump (KEEP=3, configurabile cambiando l'env del
    container in Portainer).
  - Nomi file: pcm-YYYYMMDD-HHMMSS.dump (ora UTC).

Verificare i backup:

  Portainer → Containers → pcm_backup → Console → /bin/sh →
      ls -la /backups/

Recuperare un dump sul tuo PC (senza SSH):

  Portainer → Containers → pcm_backup → Console → /bin/sh →
      base64 /backups/pcm-YYYYMMDD-HHMMSS.dump

  Copia tutto l'output dal terminale Portainer. Sul PC (PowerShell):

      $b64 = @'
      <incolla qui l'output base64>
      '@
      [IO.File]::WriteAllBytes("$HOME\Desktop\pcm.dump",
          [Convert]::FromBase64String(($b64 -replace '\s','')))

  Per dump grossi (>10 MB) la console di Portainer puo' diventare lenta.
  In quel caso usa il bundle ZIP scaricabile dalla UI admin
  ("Backup / Migration" → Download bundle): e' la stessa cosa con
  l'aggiunta degli asset (immagini, ecc.) e la scarichi via HTTPS.

Restore di un dump (sulla VM, da Portainer):

  Console di pcm_backup:
      pg_restore -h db -U <user> -d <db> -c < /backups/<file>.dump

  (Le credenziali sono gia' nelle env del container.)

NOTA limite: con KEEP=3 e 1 dump al giorno hai 3 giorni di rolling
window. Se vuoi piu' giorni, in Portainer cambia la env KEEP del
container pcm_backup (es. KEEP=7) e ricrea il container.

ALTERNATIVA RACCOMANDATA: il bundle ZIP via UI admin e' anche un
buon backup off-site. Salvalo sul tuo PC ogni tanto.


==========================================================================
7-bis) PROCEDURA STEP-BY-STEP DEL GO-LIVE
==========================================================================

Vedi DEPLOY_PROCEDURA.txt per la checklist clic-per-clic.
Ordine logico:

  PASSO 1  scarica bundle ZIP dal sito vecchio (admin → Backup/Migration)
  PASSO 2  genera SECRET_KEY, POSTGRES_PASSWORD, ADMIN_PASSWORD sul PC
  PASSO 3  spegni vecchio stack su Portainer
  PASSO 4  crea nuovo stack su Portainer (Repository mode), incolla
           le 13 env, "Deploy the stack". Aspetta che frontend_build
           esca a 0 e che caddy/backend siano "running".
  PASSO 5  applica migrazioni Alembic (console pcm_backend)
  PASSO 6  primo login + verifica HTTPS
  PASSO 7  importa pcm-bundle.zip via UI admin (wipe=true)
  PASSO 8  cambia la password admin dal pannello
  PASSO 9  smoke test (vedi sezione 9)

Niente piu' "scp dist" (sezione 6 lo fa il container di build).
Niente piu' "crontab" (sezione 7 lo fa il sidecar backup).
Niente piu' "/opt/pcm-hub" (tutto in volumi nominati Docker).

9) DOPO IL DEPLOY — SMOKE TEST
==========================================================================

Apri il sito da una macchina che NON sta sulla VM. Testa:

  [ ] La pagina pubblica home carica.
  [ ] curl https://hub.parametricomparison.unimore.it/healthz
      -> {"status":"ok"} con HTTP 200.
  [ ] curl https://hub.parametricomparison.unimore.it/docs
      -> 404 (in prod e' disabilitato).
  [ ] Login con un utente test funziona.
  [ ] Login con password sbagliata 6 volte di fila -> la 6a deve
      tornare HTTP 429 (rate-limit attivo).
  [ ] Creazione/modifica di un parametro funziona (admin).
  [ ] Compilazione di un blocco parametro per una lingua funziona.
  [ ] Il save mostra l'errore 409 se due tab modificano lo stesso blocco
      e il primo salva (test della concorrenza).
  [ ] Logout pulisce la sessione.
  [ ] In Portainer: pcm_backend "(healthy)", pcm_caddy "running",
      pcm_db "(healthy)", pcm_backup "running",
      pcm_frontend_build "exited (0)".

Se qualcosa rompe, i log si guardano da Portainer:
  Containers → pcm_backend → Logs (e idem per db, caddy, backup,
  frontend_build).
  Per i certificati Let's Encrypt: log di pcm_caddy. La riga utile e'
  "obtain successful". Se vedi "no IP for ..." o "challenge failed",
  controlla DNS e firewall (porte 80/443 da internet).


==========================================================================
10) ROTAZIONE DELLA SECRET_KEY (in caso di sospetto)
==========================================================================

Se per qualunque motivo pensi che la SECRET_KEY di prod sia stata vista
da qualcuno (collega che ha lasciato il team, screenshot per errore...):

  1. Genera una chiave nuova (sezione 2.A).
  2. Aggiorna SECRET_KEY in Portainer.
  3. Ricrea il container backend.
  4. Effetto: TUTTI i token attivi vengono invalidati. Tutti gli utenti
     dovranno rifare login. Va detto in anticipo.

La chiave compromessa nel git history vecchio (commit pre-c7df8a1) è
inutile dal momento in cui hai una chiave nuova in produzione. Non c'è
bisogno di riscrivere la history.

APPUNTI VARI
==========================================================================

- Il file .env LOCALE non va mai committato. Il .gitignore lo blocca.
- Il file .env.example serve come template per chi clona il repo.
- In dev la porta DB è esposta su 127.0.0.1:5433 (non 5432) per evitare
  conflitti con un eventuale Postgres nativo di Windows.
- In dev il frontend (vite su :5173) parla con il backend (:8000) usando
  VITE_API_URL=http://localhost:8000 (vedi frontend/api.js, fallback
  automatico in modalita' DEV). In prod il browser usa path relativi e
  Caddy proxa le API.
